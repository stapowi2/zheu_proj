## Архитектура системы

1. **Client (Frontend)**: Веб-интерфейс, который регистрирует Service Worker, запрашивает у браузера ключи push-подписки и отправляет их на бэкенд.
2. **API Server (Go)**: Принимает HTTP-запросы, валидирует JWT-токены пользователей, сохраняет заявки в БД и отправляет ID задач в redis.
3. **Redis**: Выступает в роли легковесного и быстрого брокера сообщений (очередь `notifications_queue`).
4. **Notification Worker (Go)**: Фоновый процесс, который в бесконечном цикле блокирующим образом (`BRPop`) забирает задачи из Redis, сопоставляет авторов заявок с их активными push-подписками в БД и отправляет уведомления через Web Push.

## стек

* **Язык разработки:** Go
* **База данных:** PostgreSQL (GORM в качестве ORM)
* **Брокер очередей:** Redis (go-redis/v9)
* **Доставка уведомлений:** Web Push API (библиотека `webpush-go`)
* **Контейнеризация:** Docker / Docker Compose

## 📡 API Эндпоинты

### Авторизация
Все эндпоинты, кроме раздачи статики, защищены через `AuthMiddleware` и ожидают заголовок:
`Authorization: Bearer <JWT_TOKEN>`

## 📡 API Эндпоинты

### Авторизация и пользователи
* **POST /auth/register** — Регистрация нового пользователя
* **POST /auth/login** — Аутентификация пользователя и выдача пары JWT-токенов
* **POST /auth/refresh** — Обновление протухшего токена доступа (Access Token)

### Заявки (Requests)
* **POST /requests** — Создание новой заявки жильцом `[Требуется авторизация]`
* **GET /requests/list** — Получение списка заявок `[Требуется авторизация]`
* **POST /requests/status/{id}** — Изменение статуса заявки (отправляет ID в Redis) `[Только для сотрудников]`

### Новости (News)
* **GET /news** — Получение списка всех новостей (с поддержкой кэширования через Redis)
* **POST /news/create** — Создание новой публикации `[Только для администраторов]`

### Уведомления (Notifications)
* **POST /notifications/register** — Сохранение push-подписки браузера в БД с привязкой к user_id `[Требуется авторизация]`
* **POST /notifications/unsubscribe** — Удаление подписки из базы данных `[Требуется авторизация]`

## Инструкция по тестированию пуш-уведомлений

1. Откройте браузер на странице `http://localhost:8080`.
2. Вставьте в текстовое поле актуальный JWT-токен жильца.
3. Нажмите кнопку **«Разрешить уведомления и подписаться»**. Убедитесь в панели *F12 -> Network*, что запрос к `/notifications/register` завершился со статусом `201 Created`.
4. На маке в системных настройках разрешите отправку уведомлений из браузера
5. Откройте терминал и отправьте `curl`-запрос от имени сотрудника для изменения статуса созданной заявки (например, ID = 5):

```bash
curl -X POST http://localhost:8080/requests/status/5 \
  -H "Authorization: Bearer <АКТУАЛЬНЫЙ_ТОКЕН_СОТРУДНИКА>" \
  -H "Content-Type: application/json" \
  -d '{
    "status": "В работе"
  }'
```

5. В логах контейнера `api-1` мгновенно отобразится процесс перехвата задачи воркером:
   ```text
   Поймал задачу из Redis для заявки ID: 5
   Заявка найдена в БД. Автор user_id: 1, Текущий статус: В работе
   Подписка найдена для user_id: 1
   Пуш отправлен! Статус ответа шлюза: 201 Created
   ```
6. На экране вашего компьютера всплывет системное push-уведомление с обновленным статусом вашей заявки.

7. Используйте gen_keys.go для генерации VAPID-ключей
